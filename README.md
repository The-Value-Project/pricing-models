# pricing-models

**JSON schemas and a deterministic specification for sales-led B2B pricing, deal configuration, and invoice generation.**

Part of [The Value Project](https://github.com/the-value-project) — open schemas for the full B2B commercial lifecycle.

---

## Overview

This repository contains three interoperable JSON schemas and a full specification defining how a B2B vendor's pricing is represented, how a deal is configured, and how a price is computed:

| Schema | File | Role |
|--------|------|------|
| `PricingModel` (PMS v1.4) | `schemas/pricing_model.json` | What the vendor offers |
| `DealConfiguration` | `schemas/deal_configuration.json` | What the customer buys |
| `InvoiceStatement` | `schemas/invoice_statement.json` | What the customer owes |

Together with the [Sales-Led Pricing Meta-Model Specification](./spec/pricing-meta-model-spec.md), these schemas define a **deterministic, auditable, LLM-compatible** pricing system for sales-led B2B products.

## The Three Schemas

### `PricingModel` — `schemas/pricing_model.json`

The Pricing Model Schema (PMS) is a structured, computable representation of a vendor's complete pricing. It is designed to be produced from any source (pricing pages, sales decks, contracts) by an LLM or a human modeler, and consumed by a pricing engine, configuration UI, or deal recommender.

**Top-level structure:**

```
PricingModel
├── products[]              Core product catalog
│   ├── modules[]           Individually priced capabilities (each has a price object)
│   ├── tiers[]             Plan levels — sets of included modules + credit grants
│   └── plans[]             Commitment-period options with percentage adjustments
├── bundles[]               Multi-product bundles with adjustment percentages
├── discounts[]             Available discounts (manual + strategy-driven)
├── credit_types[]          Entitlement pool definitions with consumption rules
├── credit_packages[]       Purchasable credit blocks
├── discount_strategies[]   Parameterized discount functions (step, linear, lookup)
├── price_books[]           Regional price overrides
├── variants[]              Deal-motion-specific overrides (new_logo, renewal, etc.)
├── minimum_commitments[]   Minimum spend floors
├── pricing_rules           Calculation order, discount order, rounding, policy
├── business_rules[]        Plain-text rules (hard_block | warning | info)
├── module_dependencies[]   Module dependency graph
└── pricing_relationships[] Cross-product rules
```

**Price primitives:** All pricing uses exactly one of four types: `flat`, `per_unit`, `tiered` (graduated bands), or `volume` (all units at matching band price). No free-form expressions exist anywhere in the model.

**Module → Tier → Plan relationship:**
- A **module** is an individually priced capability. Every priced item is a module.
- A **tier** is a plan level — a set of included module IDs. A tier's cost is the sum of its modules' prices.
- A **plan** is a commitment period — a percentage adjustment applied to module prices.

**Credit types and entitlements:** Any unit of value (seats, credits, API calls, tokens) is modeled as a `credit_type` with `consumption_rules` and an `overage_policy`. Entitlements are granted via `included_credit_grants` on tiers, modules, plans, and bundles — never as hardcoded fields.

**Business rules:** Every constraint is expressed as a plain-text `description` plus a `rule_type` enum for programmatic routing and an `enforcement` level (`hard_block`, `warning`, or `info`). The description alone is sufficient for a human or LLM to understand and apply the rule.

### `DealConfiguration` — `schemas/deal_configuration.json`

A customer-specific deal that references a PMS instance. Created by a salesperson via a configuration UI, or generated by an LLM given a customer value model and target price.

**Top-level structure:**

```
DealConfiguration
├── config_id / config_date
├── model_reference.model_id  References the PricingModel being configured
├── customer                  name, domain, region
├── deal_motion               new_logo | renewal | expansion | reprice
├── billing                   plan_id, contract_start_date, contract_term_months
├── product_selections[]
│   ├── product_id / tier_id
│   ├── quantities[]          credit_type_id, quantity, quantity_type, growth_schedule
│   └── credit_package_selections[]
├── bundle_selections[]
├── discount_selections[]     discount_id + manual_value (for manual discounts)
└── negotiated_overrides
    ├── price_overrides[]     fixed_price | discount_percentage | waived | capped
    ├── credit_grant_overrides[]
    └── custom_terms[]        sla_upgrade | price_lock | mfn_clause | other
```

**Quantity types:** Each quantity in a product selection has a `quantity_type` — `committed` (contractual minimum), `estimated` (forecast), `maximum` (capacity ceiling), or `starting` (initial value with a `growth_schedule`).

**Join keys:** All `product_id`, `tier_id`, `plan_id`, `credit_type_id`, `credit_package_id`, `discount_id`, and `bundle_id` values reference IDs declared in the `PricingModel`.

### `InvoiceStatement` — `schemas/invoice_statement.json`

The deterministic output of the pricing engine — a fully auditable invoice computed from a `PricingModel` + `DealConfiguration` pair.

**Top-level structure:**

```
InvoiceStatement
├── invoice_id / config_id / period / currency
├── line_items[]
│   ├── category          module_charge | credit_package | usage_overage | one_time_fee | ...
│   ├── product_id / target_id
│   ├── quantity / unit / unit_price / line_total
│   ├── graduated_detail[]   band-level breakdown for tiered/volume pricing
│   ├── computation_method   human-readable trace
│   └── source_fields[]      which PMS/config fields drove this line
├── subtotals             recurring / one_time / gross
├── adjustments[]
│   ├── adjustment_type   plan_adjustment | volume_discount | strategy_discount | ...
│   ├── amount            signed (negative = discount, positive = true-up)
│   └── source            which PMS object drove this adjustment
├── totals                total_adjustments / net_total / annualized_total / total_contract_value
├── entitlement_summary[] per credit_type: granted / committed_or_used / overage / overage_cost
└── warnings[]            error | warning | info
```

Every line item traces to `source_fields` in the model and config. Every adjustment identifies its source object. The invoice is fully reproducible from its inputs.

## The Computation Engine

The specification defines a **9-phase deterministic evaluation algorithm**:

| Phase | Name | What happens |
|-------|------|-------------|
| 0 | Validate | Check join keys, hard-block business rules, module dependencies |
| 1 | Variant | Apply deal-motion overrides |
| 2 | Price Book | Apply regional price overrides |
| 3 | Plan | Apply commitment-period adjustments |
| 4 | Line Items | Compute module + credit package charges |
| 5 | Overages | Compare usage to entitlements, compute overage costs |
| 6 | Discounts | Apply discounts in declared `discount_application_order` |
| 7 | Minimums | True up to minimum commitment floors |
| 8 | Assemble | Sum subtotals, net total, annualize, compute TCV |

The specification also defines a **reverse-solve algorithm**: given a target price, it works through available levers (discount %, plan commitment, bundle selection, quantity adjustment, tier downgrade) to suggest the configuration adjustments needed to reach the target.

Full algorithm details are in [`spec/pricing-meta-model-spec.md`](./spec/pricing-meta-model-spec.md).

## Discount Strategies

Strategy-driven discounts use named, parameterized computation functions:

| Shape type | Behavior |
|-----------|---------|
| `step_tiers` | Discount % by quantity bracket |
| `linear_ramp` | Linear interpolation from a start point |
| `piecewise_linear` | Multi-segment linear interpolation |
| `lookup_table` | Key-value lookup |

Strategy values are always positive (representing discount magnitude); the engine applies the sign. Plan adjustments use negative percentages directly.

## LLM Usage

**Building a `PricingModel`:** Provide the LLM with pricing source content (pricing page, sales deck, contract) and the PMS schema as a structured output target. The spec includes a full prompt architecture and quality checklist. Key guidance: every priced item is a module; tiers sum their modules; plans apply percentage adjustments; business rules must have self-contained plain-text descriptions.

**Generating a `DealConfiguration`:** Provide the LLM with a `PricingModel` instance and a `CustomerVariables` instance (from the [`value-models`](https://github.com/the-value-project/value-models) repo). The spec defines a mapping table from value model concepts (team size, transaction volume, budget, deal motion) to configuration fields.

All schemas use `additionalProperties: true` at the top level, supporting implementation extensions while preserving the required core structure.

## Publishing a Pricing Page

A `PricingModel` JSON instance can be published as a machine-readable pricing page — a Markdown document that wraps the JSON with structured human-readable descriptions, front matter metadata, and a pointer to the LLM interpreter reference. The page is designed to be read by both humans (for accuracy review) and LLMs (for pricing, configuration, and deal workflows).

Two documents in this repository support this:

- **[`spec/pricing-page-spec.md`](./spec/pricing-page-spec.md)** — specifies the format of a conforming pricing page, including required front matter fields, a structured human-readable description (modules table, plans table, credit types, discounts, business rules verbatim by enforcement level, key pricing policies), and the PricingModel JSON block
- **[`spec/pricing-model-llm-reference.md`](./spec/pricing-model-llm-reference.md)** — a prompt-like reference document published alongside pricing pages; instructs an LLM how to interpret the PricingModel JSON, apply the four price primitives, evaluate entitlements and overages, apply discounts in order, enforce business rules, and answer common pricing questions

The LLM reference is linked from every conforming pricing page via the `llm_ref` front matter field, so any LLM fetching a pricing page will find its interpreter instructions automatically.

## Conformance Checklist

An implementation conforms to this specification if:

- [ ] PricingModel output validates against the schema
- [ ] All price computations use only the 4 price primitives (flat, per_unit, tiered, volume)
- [ ] No free-form expressions exist in the pricing engine
- [ ] Join keys are consistent across all three schemas
- [ ] Every invoice line traces to model fields and computation method
- [ ] Business rules with `hard_block` prevent invalid configurations
- [ ] Engine is deterministic: identical inputs → identical outputs
- [ ] Entitlements use generic credit_types + credit_grants (no hardcoded fields)
- [ ] Discounts apply in declared `discount_application_order`
